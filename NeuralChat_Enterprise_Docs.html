Section 01
Project Overview
What NeuralChat does, the problem it solves, and its core capabilities.
What is NeuralChat?

NeuralChat is a full stack AI chatbot that allows authenticated users to upload private documents such as PDFs and text files, then ask natural language questions answered from the content of those documents. It uses a technique called Retrieval-Augmented Generation (RAG) to combine a local semantic search layer with a free-tier LLM.

The Problem

Standard LLMs are trained on public data with a knowledge cutoff. They have no access to private documents, internal reports, or domain-specific files. Without RAG, the only alternative is expensive fine-tuning which requires GPU time and becomes outdated the moment documents change.

Why RAG?

RAG adds a retrieval layer that searches your own documents before generating a response. It requires no training, supports instant knowledge updates by simply uploading a new file, and produces answers grounded in real source content rather than hallucinated responses.

Key Features

JWT Auth bcrypt Hashing Protected Routes PDF and TXT Upload Semantic Retrieval FAISS Vector Index Groq LLM (Free Tier) Conversation History LangChain RAG Pipeline sentence-transformers Embeddings

Request Lifecycle

1
User sends a message React sends POST /api/chat/message with the message text and a Bearer JWT token in the Authorization header.
2
FastAPI validates the token The JWT is decoded, the signature and expiry are verified, and the username is extracted from the payload.
3
Message saved to database The user message is written to the chat_messages table in SQLite with role=user and a server-generated timestamp.
4
RAG retrieval runs The query is embedded into a vector using sentence-transformers. FAISS searches for the top 4 most semantically similar document chunks.
5
LLM generates the answer LangChain builds a prompt with the retrieved chunks and conversation history, then sends it to Groq's llama3 model.
6
Response saved and returned The assistant response is saved to SQLite as role=assistant and returned to React, which renders it as a chat bubble.

Frontend Architecture
React component structure, authentication flow, and state management.

Component Structure

main.jsx
  BrowserRouter
    AuthProvider        (global auth state via Context API)
      App.jsx
        /login          -> LoginPage.jsx       (public)
        /register       -> RegisterPage.jsx    (public)
        /chat           -> ProtectedRoute
                             -> ChatPage.jsx   (JWT required)
        *               -> Redirect to /login
AuthContext

Holds token and username in React state. Reads from localStorage on page load so sessions survive refresh. The login() function writes to both localStorage and state. logout() clears both and the Axios interceptor redirects to /login on any 401 response.

ProtectedRoute

A wrapper component that checks isAuth from AuthContext. If the user is not authenticated it redirects immediately to /login using React Router's Navigate with replace=true, preventing the back-button from returning to a protected page.

Axios Service Layer (api.js)

A single Axios instance is used for all API calls. A request interceptor automatically attaches the JWT token from localStorage as a Bearer token on every outgoing request. A response interceptor catches 401 errors, clears the token, and redirects to /login. This centralizes auth logic so no component needs to handle it manually.

Section 05
RAG Pipeline Overview
How documents are processed, stored, and retrieved to power context-aware responses.
RAG Network Flow Local CPU Embeddings
Core Idea: RAG separates knowledge retrieval from response generation. The retriever finds relevant content. The LLM uses that content to write a grounded answer. Neither alone is sufficient for domain-specific Q and A.

Ingestion Pipeline (On Upload)

1
Document Loading PyPDFLoader reads PDF pages. TextLoader handles plain text files. Both return LangChain Document objects with page_content and metadata fields.
2
Chunking RecursiveCharacterTextSplitter breaks documents into 512-character chunks with 64-character overlap. The overlap prevents important sentences from being split across two chunks and losing context.
3
Embedding Each chunk is converted to a 384-dimensional vector using sentence-transformers all-MiniLM-L6-v2, running locally on CPU. Similar text produces vectors that are close in vector space.
4
Storage Vectors are stored in a FAISS index on disk. Metadata (filename, user_id, chunk count, timestamp) is saved to a JSON file alongside the index.

Query Pipeline (On Message)

1
Query Embedding The user's question is embedded using the same model, producing a vector that represents its meaning.
2
Similarity Search FAISS computes cosine similarity between the query vector and all stored chunk vectors. The top 4 most relevant chunks are returned.
3
Prompt Construction LangChain injects the 4 retrieved chunks plus the last few conversation turns into a structured prompt sent to the LLM.
4
LLM Response Groq's llama3 reads the context and generates a factual, grounded answer at temperature 0.3 for consistent results.

RAG vs Normal Chatbot

AspectNormal ChatbotNeuralChat with RAG
Knowledge sourceTraining data onlyYour uploaded documents
Private data accessNot possibleFull access via retrieval
Update knowledgeRequires retrainingUpload a new file
Accuracy on domain dataHallucinatesGrounded in source chunks
CostFine-tuning: very expensiveInference only: near zero

Database Design
SQLite schema, ORM models, and metadata storage strategy.
SQLite Database

Users Table

class User(Base):
    __tablename__ = "users"
    id               = Column(Integer, primary_key=True, index=True)
    username         = Column(String(50), unique=True, index=True)
    email            = Column(String(100), unique=True, index=True)
    hashed_password  = Column(String(255), nullable=False)
    created_at       = Column(DateTime(timezone=True), server_default=func.now())

Chat Messages Table

class ChatMessage(Base):
    __tablename__ = "chat_messages"
    id         = Column(Integer, primary_key=True, index=True)
    user_id    = Column(Integer, ForeignKey("users.id"), nullable=False)
    role       = Column(String(10), nullable=False)   # "user" or "assistant"
    content    = Column(Text, nullable=False)
    created_at = Column(DateTime(timezone=True), server_default=func.now())
Design Notes

role field: Using a single messages table with a role discriminator matches the LLM chat format standard (user/assistant), making history directly usable as conversation memory without transformation.

server_default: Timestamps are set by the database engine, not the application, ensuring consistent time across all server instances.

Vector Metadata

FAISS stores only vectors. A companion metadata.json file tracks document names, user IDs, chunk counts, and ingestion timestamps. This bridges the gap between the vector index and the UI without querying the vector store itself.

Section 07
Authentication and Security
JWT lifecycle, password hashing, and endpoint protection.
JWT Security Shield bcrypt Lock 60 min Token Expiry

JWT Token Flow

1
Login and Token Creation On successful login, python-jose creates a JWT with payload {"sub": username, "exp": now + 60 minutes}, signed with HS256 using the SECRET_KEY from the .env file.
2
Token Storage and Transport The token is returned in the login response body. React stores it in localStorage and AuthContext. Every API request attaches it as Authorization: Bearer token.
3
Validation on Protected Routes FastAPI's OAuth2PasswordBearer extracts the token. verify_token() checks the signature, expiry, and sub claim. Any failure returns a 401 immediately.
4
Expiry and Re-login After 60 minutes the token expires. The Axios response interceptor catches the 401, clears localStorage, and redirects to the login page.

Password Hashing

# Every password gets a unique random salt
hashed = bcrypt.hashpw(password.encode("utf-8")[:72], bcrypt.gensalt())

# Verification never reveals the original password
bcrypt.checkpw(submitted.encode()[:72], stored_hash.encode())
Why bcrypt? Unlike SHA-256 which is designed to be fast (billions of hashes per second), bcrypt is deliberately slow by design. Each hash takes roughly 100ms, making brute-force attacks impractical. The built-in random salt also prevents rainbow table attacks.

Security Summary

ThreatMitigation
Password theftbcrypt with unique salt per user. Plain password never stored.
Token forgeryHS256 signature. Invalid tokens rejected immediately.
Unauthorized accessAll chat routes require valid JWT via Depends().
Cross-origin attacksCORS restricted to frontend domain only.
Secrets exposureAll secrets in .env, listed in .gitignore, never hardcoded.
SQL injectionSQLAlchemy ORM uses parameterized queries throughout.
Malicious uploadsFile extension whitelist: .pdf, .txt, .md only.
Section 08
Scalability Plan
The upgrade path from local prototype to production infrastructure.
Scalable Stack Architecture
SQLite to PostgreSQL

SQLite is single-writer and unsuitable for concurrent API workers. Switching requires only a changed DATABASE_URL. SQLAlchemy ORM code is database-agnostic so no business logic changes. Add pgBouncer for connection pooling.

FAISS to Pinecone

FAISS is single-node and in-memory. For multi-server deployments, Pinecone or Qdrant provide a managed distributed vector store. LangChain's Pinecone class is a drop-in replacement. Add per-user namespaces for document isolation.

Redis Caching

Cache embedding outputs so identical queries skip the model entirely. Cache recent chat history per user to avoid repeated SQLite reads. Use TTL equal to the JWT expiry to auto-invalidate stale sessions.

Async Processing

Document ingestion currently blocks the request thread. Moving it to FastAPI BackgroundTasks or a Celery worker queue allows the API to return immediately while ingestion runs in the background.

Horizontal Scaling

FastAPI is stateless so multiple instances can run behind an Nginx load balancer. Auto-scaling groups add instances under high traffic. All state lives in external services: PostgreSQL, Redis, Pinecone.

Containerization

Docker Compose defines services for the API, Celery worker, Redis, and Postgres. For cloud deployment: AWS ECS, GCP Cloud Run, or Kubernetes for orchestration. GitHub Actions for CI/CD automation.

Production Architecture

Internet
  -> CDN (Cloudflare)
  -> Load Balancer (Nginx or AWS ALB)
      -> FastAPI Containers x3 (stateless, auto-scaled)
          -> PostgreSQL     (RDS or Cloud SQL)
          -> Redis          (ElastiCache for caching)
          -> Pinecone       (managed vector database)
          -> Celery Workers (document ingestion queue)
          -> S3             (file storage)
          -> Groq API       (LLM inference)
Section 09
Key Design Decisions
Why each technology was chosen and what the tradeoffs are.
DecisionChosenReasonWhen to Switch
API Framework FastAPI Auto OpenAPI docs, Pydantic validation, async support, native Python ML ecosystem Django for large teams needing admin and ORM. Flask for simpler single-file apps.
Frontend React + Vite Component reuse, fast HMR dev experience, industry standard, huge ecosystem Next.js when SSR or SEO is required. Vue for teams preferring opinionated structure.
RAG vs Fine-tuning RAG Zero training cost, instant updates, grounded answers, works with any LLM Fine-tuning when style, tone, or behavior must change, not just knowledge.
Vector Store FAISS (local) No server required, zero cost, fast enough for prototype, persists to disk Pinecone or Qdrant for multi-server deployments or beyond 1 million vectors.
LLM Groq free tier Free at low volume, fast inference speed, llama3 quality is strong OpenAI GPT-4 for higher quality. Ollama for fully offline. Gemini for Google infra.
Database SQLite Zero setup, file-based, perfect for single-server prototype and development PostgreSQL for any production deployment with concurrent API workers.
Auth Strategy JWT stateless No server-side session storage, scales horizontally, REST API standard Session-based auth when instant token revocation is needed. OAuth2 for social login.
Overall Philosophy: NeuralChat was built with clear layer separation so each component is independently replaceable. Swapping SQLite for PostgreSQL, FAISS for Pinecone, or Groq for OpenAI requires changes only to initialization code, not business logic. This is intentional and is the right way to build systems that need to evolve.
